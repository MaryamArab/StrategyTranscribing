Expert Programming Strategies


Welcome to the Explicit Programming Strategies research study.

When you solve problems, you might not think much about how you go about solving them, and what strategies and steps you use for. The reason is that you have used these approaches and strategies through many years of programming experiences and they change to daily routines.

In this study, you will use expert programming strategies to complete two programming tasks. Before you get started, we have some background to help explain what is a programming strategy and how you can use one.

What is a Programming Strategy?

A strategy is a high-level plan for accomplishing a task, describing a series of steps or actions to take to accomplish a goal.

Strategies are a common part of everyday life, describing steps with which to find an email in an email client, cook the perfect pizza, or tie your shoes. Programming strategies describe how software developers do similar everyday tasks, to diagnose a performance issue or a broken control.

Expert Programming Strategies

Through constantly facing tricky programming situations, expert software developers develope strategies which enable them to perform complex programming tasks more quickly and easily.

We gathered expert software developers in web development and asked them to write down a strategy on how they would approach a challenging programming task.

In this study, you’ll have an opportunity to learn from two web development experts and use the strategies they wrote, step by step, in two programming tasks, enabling you to work in the same way the expert developer would work.

Roboto - A language for describing strategy

To help Experts write down strategies in a way that other developers can use, we’ve built a language for describing strategies called Roboto. This is a simple notation for describing procedures. Most of it is natural language, but there are a few parts that have a concrete meaning. Let's walk through a few short examples to illustrate its syntax.

Roboto language constructs

Roboto has several common features from imperative languages, including:

  1. Preconditions
  2. Statements
    • Actions
    • Definitions
    • Conditionals
    • Loops
    • Returns
    • Calls
  3. Variables

To understand how each of these is being used in writing down a strategy in Roboto, we’ll first take a look at a complete example of a strategy and then take a look at each one by one.

A sample strategy with Roboto for merging two Github branches

The text on the right is an example Roboto strategy. It describes how to merge two branches of code, including fixing any conflicts which may arise.

The strategy is started by 'Strategy' keyword, followed by the name of the strategy. In the parenthesis followed by strategy name, you can define variables that you might need to use in the strategy statements steps.

Each line in the strategy body represents a statement. A strategy consists of a set of Statements which is represented as a sequence of steps which should be followed by a user. There are multiple different statement types which will be described in next sections.

Precondition

Preconditions express requirements about what the developer using the strategy should first know or have access to.

Required tools and Environment: Consider what resources are important to your strategy. Does it require that there be a test suite? A breakpoint debugger? A detailed log file?

Required knowledge: What level of expertise should the strategy user have? What tools, languages, or concepts should they be familiar with before using your strategy?

Context: Write your strategy for a specific context, such as a defect you debugged yesterday. Describe your application domain: embedded development in C, front end development in React, desktop Java application, etc.

Available resources: For your case, consider what resources are important to your strategy. Does it require that there be a test suite? A breakpoint debugger? A detailed log file?

Specific Programming Knowledge: What programming languages are required to be familiar with

Expertise level: What level of expertise the user of strategy should have

Statement

Each statement in strategy describes what the user should do next.

It could be performing specified action, gathering information, or checking some parameters, fault, or situation in the program.

Statements in Robot can be one of six types: Action, Definition, Conditional, Loop, Return, Call , which we describe in detail next.

Additional details about each statement can be added through a comment, indicated through a # tag. These comments should be added exactly before the statement description. In the image in the right, comments are shown in blue color.

Comments are not mandatory. If you think that you cannot describe all the details in a strategy statement, or you feel that not all users need detailed or basic information, you can add it as a comment of extra information before each statement.

Try not to repeat the same text as statement in the comments, as it will not add any further information.

Actions

Action statements describe something that the user should do in the world.

Look at the highlighted statements in the strategy. There is no predefined syntax for writing an Action statement.

They are mainly in the form of an imperative sentence, which ask user to perform some action in the world, code, tool, or environment.

Definition

Definition statement always start with 'SET' keyword, followed by a variable definition, followed by "TO" keyword and a sentence which describe what the defined variable should be assigned to.

You can see a sample of Definition statement highlighted in the GitMerge strategy

Variable

Variables make it easier for the strategy user to keep track of things, such as remembering lines of code or functions to return to.

Variables may be either a primitive or a list. For instance, ‘conflicted files’ in the following example indicates a list of files which have a conflict, which subsequent statements may then explicitly refer to. The helps reduce what the strategy user has to remember, making it easier for them to correctly follow the strategy.

For instance, 'conflictedFiles' in the highlightes statement is a variable records a set of files that have a conflict, which subsequent statements may then refer to explicitly. This helps reduce memorization burden.

Conditionals

A conditional asks the strategy user to check if a condition is true in the world and then branch to different statements depending on the outcome.

Conditional statements stat with "IF" keyword, followed by a query which ask user to check if a condition is true or false in the program or environment.

Each branch in a Conditional statement consist of a set of statements to follow.

Loops

Loops indicate a sequence of statements that the strategy user should potentially take more than once.

In Roboto, loops take the form of a FOR EACH loop, iterating over a set of items (e.g., files, breakpoints, messages output by a tool) to take a set of actions for each.

The items to loop through should be defined in the previous statements as an 'Identifier'.

In this strategy, the highlighted statement shows a Loop statement which iterates through the defined variable, 'conflictedFiles', in its preceding statement.

This strategy will execute the statements followed by the Foreach statement for all the 'files' being set in 'conflictedFiles'.

Return

Returns indicate returning a variable, Call statement, or Action statement.

Call

Call statements start with 'Do' keyword following by Sub-Strategy.

They are used for making nested strategies to prevent them from being disorganized, complex, and long.

Sub-Strategy

To organize long and complex strategies, strategies can be separated into sub-strategies. A sub-strategy is a strategy called by another strategy. For example, strategy fixConflict is a sample of a sub-strategy which is being used and called in mergeGithubBranches strategy above.

STRATEGY fixConflict(conflictedFile)
Open the conflictedFile with your favorite text editor
# To find a line with conflict in your file, search the file for the conflict marker <<<<<<< HEAD
SET
'line'
TO a line number that has conflict
# until there is no line with <<<<<<< HEAD tag in the file
UNTIL the file has no lines of conflict
# You'll see the changes from the master branch after the line <<<<<<< HEAD
# Next youll see ======= which divides 2 merged lines followed by >>>>>>> SECOND branch name
# Decide if you want to keep only your branch's changes, keep only the other branch's changes, or make a brand new change, and edit the file by removing unwanted lines and save the file
# Also remove >>>>>>Head & ====== from the lines and make sure that the file has proper code syntax.
Edit the file following the description
SET
'line'
TO the next line number that has conflict
Return nothing

Warning: Self-regulation

When doing a programming task, you may have a strategy that you’ve always used in the past. Perhaps, when faced with a null pointer exception, the first thing you always do is set a breakpoint and run the code in the debugger. To successfully use and learn a strategy, you will need to stop whatever you would normally do and instead do the steps exactly as described in the strategy. In this way, you will be able to approach the problem just as an expert developer would approach the same problem.
