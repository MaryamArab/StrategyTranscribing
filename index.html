Welcome to the research study of Explicit Programming Strategies.

Software developers solve a diverse and wide range of problems. While software engineering research often focuses on tools to support this problem solving, the strategies that developers use to solve problems are at least as important as the tools they use.

In this study, we are asking developers like you to write down the strategies you use to solve problems, so that others can benefit from them.

We start with defining what we mean by Programming Strategy, then we will go through a set of tutorial steps which describe the syntax of a language for describing strategies expert developers used in their everyday development life. This language is called Roboto and a language for writing down programming strategies.

Finally, you will be asked to write down a strategy you have used in a recent programming task.

What is a Programming Strategy?

A strategy is a high-level plan for accomplishing a task, describing a series of steps or actions to take to accomplish a goal. Strategies are a common part of everyday life, describing steps with which to find an email in an email client, cook the perfect pizza, or tie your shoes. Programming strategies describe how software developers do similar everyday tasks, like different ways to debug a defect or go about software design. Some people debug by gathering lots of data about execution, then inspecting it, while others generate lots of hypotheses, testing them one by one.




Roboto language constructs

Roboto has several common features from imperative languages, including:

  1. Preconditions
  2. Statements
  3. Conditionals
  4. Actions
  5. Loops
  6. Variables

Statement

Action

Actions describe something that the user should do in the world.


Conditional

A conditional asks the user to check if a condition is true in the world and then branch to different statements depending on the outcome.


Loops

Loop helps in tracking set of steps the user should take

How to write a strategy.

Define strategy in a sequence of steps. Describe your strategy in the form of sequence of steps a developer should follow and do some action.


Describe required knowledge and tool for using your strategy. Describe what are the preconditions a developer should have to be able to use your strategy.


Use comments to elaborate. Each step in a strategy should be clear and concise, but you can use comments to help developers who may not fully know how to execute a step.Include your comments before each statement after # tag.


Avoid wasted work. Structure your strategy in a way that minimizes how much work a developer has to do


Include explicit restarts. Some strategies fail; include these failure points in your strategy, guiding developers either back to the start of a strategy, or to use a different strategy.


Include rationale. A strategy should include an overall rationale for its approach, as well as rationale for how each step achieves the high level approach. This helps motivate a developer to follow the strategy reliably.


Encourage externalization. For any information a developer will need to reference and reuse, encourage them to capture it in a variable. This ensures they don’t lose it, but also that it’s easy to reference. It also allows the data to be used in other parts of a strategy.
