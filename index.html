Programming Strategies


Welcome to the research study of Explicit Programming Strategies.

Software developers solve a diverse and wide range of problems. While software engineering research often focuses on tools to support this problem solving, the strategies that developers use to solve problems are at least as important as the tools they use.

In this study, we are looking for strategies expert developers use in their everyday programming tasks.

We start with defining what we mean by Programming Strategy, then we will go through a set of tutorial steps which describe the syntax of a language for describing strategies expert developers used in their everyday development life. This language is called Roboto and a language for writing down programming strategies.

Finally, you will be asked to write down a strategy you have used in a recent programming task.

What is a Programming Strategy?

A strategy is a high-level plan for accomplishing a task, describing a series of steps or actions to take to accomplish a goal. Strategies are a common part of everyday life, describing steps with which to find an email in an email client, cook the perfect pizza, or tie your shoes. Programming strategies describe how software developers do similar everyday tasks, like different ways to debug a defect or go about software design.




Roboto language constructs

Roboto has several common features from imperative languages, including:

  1. Preconditions
  2. Statements
  3. Conditionals
  4. Actions
  5. Loops
  6. Variables

SubStrategy

To organize long and complex strategies, strategies can be separated into substrategies. A sub-strategy is a strategy called by another strategy. For example, strategy fixConflict is a sample of a sub-strategy which is being used and called in mergGithubBranches strategy.



Precondition

Preconditions are requirements for using the strategy to fix the defect. They are the required resources, knowledge, skill, context , and any kind of requirements for using the strategy to fix the defect:
Context: Write your strategy for a specific context, such as a defect you debugged yesterday. Describe your application domain: embedded development in C, front end development in React, desktop Java application, etc.
Available resources: For your case, consider what resources are important to your strategy. Does it require that there be a test suite? A breakpoint debugger? A detailed log file?
Specific Programming Knowledge: What programming languages are required to be familiar with
Expertise level: what level of expertise the user of strategy should have


Statement

Each step in strategy definition is a statement describing what should the user do in next step

More description and clarification about the statement can be added before each statement after a # tag

Each statement is one of the following:

  1. ACTION
  2. CONDITIONAL
  3. LOOP
  4. CALL


Variable

Variables help users lower the burden of remembering, such as remembering lines of code or identifiers to return to. They can be defined as primitives and lists.

For instance, 'conflictedFiles' in the following is a variable records a set of files that have a conflict, which subsequent statements may then refer to explicitly. This helps reduce memorization burden.



Conditional

A conditional asks the user to check if a condition is true in the world and then branch to different statements depending on the outcome.


Action

Actions describe something that the user should do in the world.


Loops

Loop helps in tracking set of steps the user should take
Could be used to iterate over any set of similar actions the user should take

Guidelines for writing strategy

Imagine a novice developer came to you with this type of defect. Imagine they did not have any similar experience so far. Describe the best strategy you can offer this developer and the steps you'd take to debug this defect, being sure to include sufficient details.

Use comments to elaborate. Each step in a strategy should be clear and concise, but you can use comments to help developers who may not fully know how to execute a step.Include your comments before each statement after # tag.

Avoid wasted work. Structure your strategy in a way that minimizes how much work a developer has to do

Include explicit restarts. Some strategies fail; include these failure points in your strategy, guiding developers either back to the start of a strategy, or to use a different strategy.

Include rationale. A strategy should include an overall rationale for its approach, as well as rationale for how each step achieves the high level approach. This helps motivate a developer to follow the strategy reliably.

Encourage externalization. For any information a developer will need to reference and reuse, encourage them to capture it in a variable. This ensures they don’t lose it, but also that it’s easy to reference. It also allows the data to be used in other parts of a strategy.
